# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`@forestadmin/datasource-sequelize` adapts an existing Sequelize ORM instance into a Forest Admin datasource. Given a configured `Sequelize` connection (with models already defined), it exposes each model as a Forest Admin collection. It is the engine behind sibling datasources `datasource-sql` (which auto-builds the models from a DB URI) and `datasource-replica`; it is rarely consumed directly.

## Architecture

Entry point is `createSequelizeDataSource(connection, options)` (`index.ts`), returning a `DataSourceFactory` consumed by the agent.

- `SequelizeDataSource` (`datasource.ts`) extends `BaseDataSource` from `@forestadmin/datasource-toolkit`. It iterates `sequelize.models` and wraps each in a `SequelizeCollection`. Models are sorted by name to keep schema order stable. `executeNativeQuery` powers the "live query" feature, enabled per-connection via `options.liveQueryConnections`.
- `SequelizeCollection` (`collection.ts`) extends `BaseCollection` and implements `create/list/update/delete/aggregate`. It converts the Sequelize model into a Forest schema (`ModelToCollectionSchemaConverter`) and delegates query translation to two helpers built in its constructor: `QueryConverter` and `AggregationUtils`.
- The core job of this package is **translating the toolkit's database-agnostic primitives into Sequelize**:
  - `QueryConverter` turns a `ConditionTree` into a Sequelize `WhereOptions` (per-operator `Op.*` mapping), a `Projection` into nested `include` options, and a `Sort` into `order`. Much of the per-operator logic branches on SQL `dialect` (postgres/mysql/mariadb/mssql/sqlite) because `LIKE`/case-sensitivity/array semantics differ.
  - `ModelToCollectionSchemaConverter` + `TypeConverter` map Sequelize `DataTypes` → Forest `ColumnType`, derive the allowed `filterOperators` per type, and translate Sequelize validators into Forest validations and associations into Forest relation schemas.
  - `Serializer` normalizes raw rows after a query (notably `Date` → ISO string, `DATEONLY` → `YYYY-MM-DD`).

## Gotchas

- **Never import the bundled `sequelize` at runtime.** `sequelize` is a `peerDependency`; the customer's installed version is the source of truth. Import only *types* from `sequelize`, and reach `col`/`fn`/`where`/`query` through the customer's instance (`model.sequelize.*`). The bundled `sequelize` in `devDependencies` exists only for tests.
- `type-overrides.ts` re-declares Sequelize private fields (e.g. `_autoGenerated`, `BelongsToMany.through.model`) the package relies on but Sequelize does not export — extend these `*Ext` interfaces rather than casting inline.
- `update`/`delete` cannot use Sequelize `include`, so filters that span relations are resolved in two steps via `getWhereFromConditionTreeToByPassInclude` (first `findAll` the matching PKs, then operate on those PKs).
- `list` deliberately adds each collection's PKs to the projection (`projection.withPks`) so Sequelize doesn't prune `include` branches with empty `attributes`, then strips them back out with `projection.apply`.
- Virtual fields (`DataTypes.VIRTUAL`) are not supported and are filtered out of `create` results.
- Tests run against real databases. `docker-compose.yml` provides postgres16/mysql8/mssql2022/mariadb11 containers on non-standard ports; dialect-specific behavior in `QueryConverter` must be exercised across them.
